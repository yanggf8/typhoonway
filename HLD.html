1. System overview

Typhoon is a single Rust binary that observes an operator's work with an LLM agent, mines that work for repeated patterns, and routes high-value patterns through a manual forge loop that produces reusable command-line tools. Typhoon writes feature requests; an external forge writes code; the operator ratifies. The runtime never synthesizes source itself.

The single-binary form is deliberate, but it does not mean a single process. One compiled artifact gives one install path, one upgrade story, one audit surface, and no client/server version skew — the proposal flow run by an interactive command (typhoon tool propose approve) is the same code path the scheduler daemon eventually feeds. The subcommands share too much code (dream phases, signal capture, sandbox checks, proposal flow) for splitting into multiple binaries to pay for itself in a single-user tool. This follows the git / docker / cargo model: one binary, many subcommands, daemon and one-shot modes side by side.

At runtime, multiple processes of that binary coexist, divided into two planes by whether they go through core.

The use plane covers every process whose work counts as Typhoon being used. Three sources feed into core. The Telegram gateway (typhoon gateway --telegram) is a long-poll daemon that hosts a chat-driven agent loop and forwards normalized requests to core; this is the canonical "gateway" — a long-running channel adapter. Future channels (Slack, Discord, web hooks) would add new gateway processes alongside it without changing core. The external-agent channel needs no gateway: each agent action is a one-shot subcommand invocation (typhoon signal record to report a tool call and outcome, typhoon memory query to pull retrieval context) that enters core directly through Typhoon's normal CLI dispatch. The scheduler (typhoon cron) is a general scheduler daemon that fires registered targets directly into core — each entry is a cron expression plus a target, where the target is any typhoon subcommand or any tool on PATH. Core, invoked from any of these three sources, dispatches the actual work — calling the frontier LLM (Telegram-gateway path), spawning a tool subprocess, reading the memory store — and tells the signal recorder to persist a signal row for each invocation. The recorder is the only thing that touches the signal store. This is why scheduled triggers flow through core: a scheduled deploy-status run is recorded exactly the way a Telegram-driven one would be, and dream can therefore learn from scheduled use.

The management plane bypasses core and operates directly on state. The unified tool manager (typhoon tool, covering both proposal-flow ingestion and lifecycle operations on the tool registry) is here. Dream (typhoon dream, a standalone file-locked batch job that drains signals into proposals and memories) is here. The watchdog (typhoon watchdog) is here too — a standalone supervisor whose only responsibility is to observe the liveness of the other two long-running daemons (gateway and scheduler) and surface failures to the operator. Everything else (DB reachability, stale locks, dream-cycle health, queue depths) is intentionally out of scope at this level and is left to the components themselves or to operator-driven inspection. Smaller operator one-shots (typhoon config, typhoon sql, typhoon dream stats) exist as CLI utilities for direct state inspection but are not architectural components in their own right. None of these generate signals — they are meta-operations on the registry, the config, the signal store itself. The scheduler dispatches differently depending on target. Use-plane targets (a tool on PATH, a user-facing typhoon subcommand) go through core and are recorded as signals exactly like Telegram-driven invocations — that is the whole point of routing scheduled use through core. Management-plane targets (the seeded typhoon dream entry, a periodic typhoon tool sync for janitor sweeps) are spawned by the scheduler directly without going through core, and produce no signals; their own operational tables (dream_runs, etc.) are how scheduled management activity is observed.

All processes share state through TursoDB transactions and filesystem locks regardless of plane. Process roles and concurrency rules are detailed in §2.3.

Typhoon's own command surface is agent-first. Every subcommand accepts and emits JSON, exits with structured codes, and is safe to compose into pipelines. An interactive operator and an external agent CLI invoke the same commands through the same paths — there is no human-only or agent-only mode. This is the precondition that makes the external-agent channel work without a separate API surface — the agent's one-shot subcommand invocations are the same calls the operator could type by hand.

This document uses these terms consistently. Typhoon's CLI is the typhoon binary's own command surface. Tools are the forged and operator-promoted artifacts installed into ~/.typhoon/bin/; "tool" subsumes both dream-forged CLIs and hand-written scripts the operator has promoted — both live in the same registry under the same rules. The tool manager is the typhoon tool subcommand namespace that ingests and operates on the tool registry. Typhoon core is the in-binary module that dispatches every "use" invocation. The signal recorder is a separate component that persists signals to the signal store and is driven exclusively by core. A gateway is a long-running channel adapter — a daemon that listens on one external channel (Telegram today; future Slack, Discord, web hooks) and feeds normalized requests to core. The external-agent channel does not need a gateway because each agent action is a fresh one-shot typhoon subcommand invocation that enters core directly. The watchdog is a standalone supervisor process whose sole job is to observe the liveness of the other two long-running daemons — the gateway and the scheduler — and surface failures.

Identity terms. A canonical user is the stable per-person identity inside Typhoon, referenced by canonical_user_id. Channels are external; canonical users are internal; a binding is a verified mapping from a channel-specific identity (e.g., a Telegram peer + bot account) to a canonical user. There are two roles: the operator ratifies proposals, manages tools, and runs administrative commands; a user sends signals and consumes memory and tools. v0.1 has exactly one operator — the deploying party, seeded at init — and zero or more additional users who join via channel binding.

Three architectural principles run through everything below. First, tools are the artifact — the tool registry (TursoDB rows + binaries in ~/.typhoon/bin/, kept atomically consistent) is the system's central artifact store, and every other piece of the lifecycle exists to grow it safely. The registry is fronted by a shared access module — a library, not a process — so both core (for runtime tool lookup during dispatch) and the tool manager CLI (for operator-driven CRUD) read and write through one place; schema knowledge and registry invariants live in this module rather than being re-implemented per consumer. Second, core dispatches; the recorder records; only core can drive the recorder — every process that uses Typhoon (the Telegram-channel gateway, the external-agent endpoints, scheduled triggers) routes through core, and only core tells the signal recorder to write. The recorder is a separate component, the sole writer to the signal store; gateways and the scheduler adapt or trigger but never invoke the recorder directly. Anything that bypasses core produces no signal. This split — use plane through core, management plane around it — is the structuring rule for the runtime and shows up directly in §2.

Third, tools are shared; per-user data is isolated. A single tool registry serves every canonical user — when one user's signals motivate a tool, every user benefits once the operator approves. Signals, memory, and soul (per-user config) are scoped per canonical_user_id; cross-user reads are a privacy bug. Storage shape is one shared store per concept (one signals table, one memories table, one config table), with rows tagged by canonical_user_id; per-user isolation is enforced by L2 library APIs that take the user ID and filter every read and write — no per-user queues or partitioned tables. Dream is the deliberate exception: its signal scan reads across all users so cross-user pattern overlap can become evidence for a shared tool. The operator role is the gate on tool ratification and registry mutations; user-role processes contribute signals and consume memory and tools, nothing more. This per-user / shared split lives in S5's data-access libraries (§2.4) and is checked at gateway-resolution and management-command entry; the rest of the architecture is identity-agnostic.

The tool manager is firmly on the management plane: it operates directly on the registry and never goes through core, because managing tools is meta, not use. Dream is also management plane — it is a standalone batch process that reads accumulated signals and writes proposals and memories. The two are complements: core feeds the signal table by recording use; dream drains it into the registry's proposal queue; the tool manager ratifies and operates on the result.

Typhoon's lifecycle moves through six stages. Stages 2 through 6 form a continuous loop in steady state; stage 1 runs once per machine.

Stage 1 — Cold start

The operator initialises a Typhoon instance against a TursoDB account. Schema migrations and seed run together and are idempotent: re-running the bootstrap on an already-initialised database is a no-op. The seed creates the operator as the first canonical user and marks them with the operator role; this is the only canonical user that exists at end of bootstrap. A cloud replica may be linked at this point or later. After bootstrap the runtime is dormant — no daemon runs, no signals exist. One Typhoon instance owns one TursoDB database that serves multiple users (the operator plus any users who later join via channel binding); the optional cloud replica is for read access from other devices, not for running a second writer instance against the same database. Multi-instance sync is not supported.

Stage 2 — Online interaction

The operator opens a session through one of two channels. Each channel has its own gateway process, but both gateways feed into the same core, so core sees a uniform interface regardless of channel.

Telegram channel (typhoon gateway --telegram). The Telegram gateway is a long-poll daemon that hosts the agent loop. On every incoming message, the gateway resolves the Telegram peer (channel + bot account + peer ID) to a canonical user via the binding store. A bound peer flows straight into the loop. An unbound peer goes through onboarding — the binding flow's exact shape is a DLD detail; the HLD only requires that a message is never accepted from an unverified peer. Once the canonical user is known, every step routes through core: the gateway asks core for that user's relevant memories, sends the prompt plus available tools to the frontier LLM, hands any tool calls the LLM emits to core for execution (shell, filesystem, tools on PATH), feeds the results back to the LLM, loops until the LLM stops, and posts the final reply to the chat. As core dispatches each invocation, it tells the signal recorder to write a signal row tagged with that user's canonical_user_id. In this path Typhoon is the agent runtime; core is the dispatcher under it.

External-agent channel (Claude Code, Cursor, Codex, …). The operator runs the agent on their machine; the agent hosts its own LLM client, its own tool dispatcher, and its own conversation loop. Typhoon never sees the frontier LLM. The agent uses Typhoon by invoking subcommands directly — typhoon memory query for retrieval context, typhoon signal record to report a tool call and outcome. In v0.1 only the operator uses this channel, so each invocation defaults to the operator's canonical_user_id with no explicit identity argument; when a second user wants this channel later, an explicit --user flag will be added. Each invocation is a one-shot process that enters core through Typhoon's normal CLI dispatch; core then drives the recorder (or reads memory) just as it would for the Telegram-channel path. A signal row is a signal row regardless of which source delivered the call. In this path the agent runtime lives outside Typhoon, but channel data still flows into core and through the recorder into the signal store.

This pattern generalizes — adding a future channel (Slack, Discord, web hooks) means writing a new gateway process that drives core through the same interface. Core does not change, the signal model does not change, and the management plane is unaffected.

After a turn closes (in either channel), a success flag is stamped onto the signal chain by a simple post-hoc rule: clean exit plus no corrective next turn. Signals accumulate; nothing else happens online.

Stage 3 — Dream

The dream cycle is triggered by the scheduler firing the seeded entry (scheduler→core→spawn dream) or by the operator invoking it directly. Once dream starts, it runs on the management plane: it operates directly on state without going back through core, since reading signals and writing proposals is meta and would loop if it generated more signals. A file lock guarantees a single dream runs at a time. Three phases run sequentially against the accumulated signals: a light phase deduplicates and sorts; a REM phase clusters successful signal chains, detects repeated workflows, and checks each candidate against the existing tool registry to flag collisions and replacement candidates; a deep phase, using a cheap batch LLM, emits three kinds of output — memories that land directly in the memory store, tool feature requests that enter the proposal queue with status awaiting_forge, and soul proposals that enter their own queue. Stale signals that never graduated are pruned. Dream is the canonical scheduled job in v0.1, but it shares the scheduler mechanism with any other entry the operator adds.

Stage 4 — Forge (out-of-band)

The operator reads awaiting_forge requests via the tool manager (typhoon tool propose list / show) and drives an external forge to produce source code that satisfies the request. This step is deliberately outside Typhoon's control: Typhoon does not invoke the forge, does not constrain the language, and does not judge the source's correctness. The operator submits the resulting source (with any correctness argument the forge produced) back through the tool manager (typhoon tool propose submit). On submission, Typhoon attaches the source, runs a platform-contract sandbox check that verifies the request's tier claim is honest, and runs a regex lint that rejects hardcoded absolute paths. A failed tier claim downgrades the tier silently — the source is preserved and review intensity rises. A failed path lint is fatal and bounces the proposal. On pass the proposal advances to awaiting_user.

Stage 5 — Ratification

The operator reviews the request, the delivery, and the forge's correctness argument together via the tool manager (typhoon tool propose approve | reject). Approval is one atomic transaction: the registry row is inserted, the tool binary is written into the install directory, a seed memory is written so the agent can discover the new tool, and — for a replacement — the predecessor is moved into .history/. Either everything succeeds or nothing does; there is no partial install. Soul proposals follow the same pattern with one substitution: the artifact is a config value rather than a tool binary, and the atomic write updates the row instead of the filesystem. A pattern rejected three times stops dream from re-proposing it.

Stage 6 — Steady use and dream-driven evolution

Once a tool is installed, every user benefits from it — the registry is shared. Memory retrieval surfaces the tool in any user's subsequent sessions; the agent invokes it as an ordinary subprocess on PATH. Each invocation increments the registry's aggregate use count and (on clean exit) the success count, with a per-user breakdown also recorded so dream can detect tools that work for one user but fail for another. These counters feed back into dream's next pass: a rising error rate produces a replacement proposal that goes through the same forge-then-ratify path as the original; prolonged disuse (default 30 days) produces a deprecation proposal that, once approved, moves the tool to disabled and archives the binary. Evolution is therefore symmetric with creation — same proposal queue, same operator approval, same atomic install.

Stage 7 — Operator-driven tool management and inspection

The tool manager (typhoon tool) is the operator's direct, agent-independent control surface for the registry, covering the entire lifecycle through one unified namespace. Every typhoon tool subcommand requires the operator role; user-role processes can use installed tools but cannot mutate the registry. The ingestion half (tool propose list / show / submit / approve / reject) is what stages 4 and 5 invoke. The operations half acts on already-installed tools: read-only listing, inspection, and lineage (tool list with filters by tier or staleness, tool show for source/stats/origin, tool diff and tool history for version comparison) and state-changing commands (tool disable takes a tool off PATH while keeping the registry row; tool enable reverses it; tool rollback restores the previous version from .history/; tool delete marks the registry row deleted (retained for lineage and tool history) and archives the binary; tool purge hard-deletes the registry row and clears .history/; tool promote adopts a hand-written script the operator authored outside the dream loop and adds it as approved_by='user' with no origin proposal; tool check-deps scans declared external dependencies across the registry; tool sync rebuilds tool binaries from registry source on a second machine). Every state change is atomic against the registry's metadata-and-binary pair. Beyond the registry, broader inspection paths are read-only: typhoon sql for SELECT queries, typhoon dream stats for the proposal funnel, and the watchdog for daemon-liveness status. typhoon sql accepts SELECT only; writes (INSERT / UPDATE / DELETE / DDL) are hard-rejected so the inspection path can never accidentally mutate state.

2. Architecture

Purpose: describe Typhoon's structure from three complementary angles, then group the pieces into subsystems that partition the DLD.

2.1 Layer view

The layer view shows the static stack of code dependencies inside the Typhoon binary. The single rule: dependencies point downward. Code at any layer may call any layer below it but never invokes — or imports from — a layer above. Same-layer dependencies are allowed but kept minimal. The runtime topology of these layers (which processes, which planes, which concurrency) is the subject of §2.3; here we are concerned only with what code can reach what.

L4 — Subcommand entry points. One thin handler per typhoon <verb> subcommand. Parses args, validates inputs, dispatches to a single L3 module. No domain logic lives here; an L4 handler that grows business rules is a smell that the rules belong in L3.

L3 — Runtime modules. The workers that do something at runtime: core (dispatcher), signal recorder (driven only by core), the gateway implementation (Telegram-channel agent loop), the cron scheduler, the dream batch, the tool manager, the watchdog. L3 modules talk to each other only through narrow contracts — for example, core is the only L3 module that may invoke the recorder.

L2 — Data access layer (domain libraries). The layer that owns Typhoon's data semantics. One library per concept: tool registry (the shared module from §1, used by both core and the tool manager), memory store, signal store, proposal queue (covering both cli_proposals and soul_proposals), config, and the identity binding store (channel identity → canonical_user_id, plus role lookup). Each library encapsulates schema knowledge and the invariants for its concept — atomicity of metadata-and-binary in the registry, success tagging in the signal store, Top-K + similarity bounds in memory retrieval, type validation and clamping in config, verified-binding semantics in identity. The shared/per-user split lives here too: tool registry is unscoped (shared); memory store, signal store, and the per-user slice of config take a canonical_user_id on every API; identity bindings own the operator-vs-user role check that gates management-plane operations. Multiple L3 modules consume the same library so those invariants live in one place.

L1 — Adapters. The only layer allowed to talk to the outside world: TursoDB / libSQL client (with transaction primitives), filesystem operations (atomic install to ~/.typhoon/bin/, archival to .history/, lock files), LLM clients (frontier and dream), Telegram client, and the sandbox + path-lint helpers used during forge submission. L1 adapters know nothing of Typhoon's data model — schema, lineage, atomicity rules, success tagging, retrieval bounds all live one layer up in L2. In v0.1 the dependency is wired directly (L2 imports L1 concrete clients); the data semantics still live in L2, so swapping a concrete adapter (e.g., cloud-replicated TursoDB ↔ a plain local libSQL file) requires only that the new adapter expose the same surface — no L2 rewrite, no L3 awareness. A full Ports-and-Adapters inversion (L1 implementing traits owned by L2) is deferred until adapter swapping becomes a real requirement; for a single-user v0.1 it would be boilerplate without payoff.

L0 — Foundation. Cross-cutting primitives any layer may use: logging, error types, time and clock. Drawn as L0 so the dependency rule is unambiguous; in practice these are ambient and the diagram's dotted "also" arrows from L3 and L2 are reminders, not exhaustive.

Three boundaries are load-bearing. First, L4 stays thin — entry points never contain logic; they dispatch. Second, L3 never reaches L1 directly — runtime modules go through L2 libraries, which is what makes registry / signal / memory invariants enforceable in one place. Third, L2 never imports L3 — a library reaching into a worker would invert the stack and create cycles. The use-plane / management-plane split (§2.3) is a runtime concern and is invisible at this layer; both planes are made of the same L3 modules calling the same L2 libraries.

2.2 Logical view (modules)

The logical view enumerates the modules — function groups — that make up Typhoon. Each module has one job, named clearly, with explicit consumers and producers. The view spans only the two layers where Typhoon's code lives: L3 (runtime modules — workers and dispatch) and L2 (the data access layer). L4 entry points are thin handlers that dispatch into L3, so they are not modules in their own right; L1 adapters and L0 foundation are infrastructure that L2 uses, also not enumerated here. The function-to-module matrix and the box-and-arrow diagram below let a reader trace any capability to the modules that implement it.

2.3 Process view

The process view is about runtime topology: what OS processes exist, which run continuously, which come and go, what they share, and how concurrency is bounded. It complements §2.1 (static dependencies) and §2.2 (functional modules) — the same modules can run in different processes and concurrency contexts depending on which subcommand is invoked.

Coordination — transactions and the dream lock

Two mechanisms hold the runtime together: SQL transactions for atomic multi-step state changes, and a filesystem lock for the one operation that must not run twice.

Transactions. Every state change that affects more than one row, or more than one storage medium (DB + filesystem), is wrapped in BEGIN IMMEDIATE … COMMIT with rollback on any failure. The atomic operations:

OperationWhat is committed atomically
Tool approval (new)registry insert + binary write to ~/.typhoon/bin/ + seed memory write + proposal status → approved
Tool approval (replacement)old binary → .history/<name>.<ts> + new binary install + registry lineage update + seed memory update + proposal status → approved
Tool rollbackcurrent binary → .history/ + previous version restored from .history/ + registry update
Tool disable / enable / delete / purgeregistry status mutation + binary move (or restore, or hard delete) consistent with the new status
Soul approvalconfig row update + proposal status → approved
Forge submissionsource attach + tier downgrade if sandbox check disagrees + proposal status → awaiting_user

Either everything in a transaction succeeds or none of it does — there is no half-installed tool, no orphan registry row, no config change without a matching proposal close-out. Because SQL transactions cannot roll back filesystem writes, atomicity across the DB+FS pair is implemented by writing tool binaries to ~/.typhoon/bin/.staging/<name>.<txn-id> first, then atomic rename(2) into ~/.typhoon/bin/<name> only after COMMIT succeeds. On commit failure the staging file is unlinked. A crash between commit and rename leaves the registry claiming a tool is active while the binary is missing from PATH; the tool sync janitor handles all post-crash reconciliation by walking three pairs and repairing any mismatch: (a) staging files with no live registry row → unlink; (b) registry rows marked active with a binary at the expected path → no-op; (c) registry rows marked active with no binary at the expected path → re-rename from any matching staging file if one exists, otherwise mark the row as broken and surface to the operator. The staging+rename pattern is the contract; the janitor closes every crash window the contract opens.

Dream lock. The dream batch acquires ~/.typhoon/dream.lock via filesystem locking before doing any work. A second invocation while the first holds the lock fails fast with a lock-error exit code. This guards both operator-invoked dream and cron-fired dream — the lock does not care who tried to start the second instance. The lock is process-scoped (released on process exit, even on crash), so a stale lock from a dead PID resolves itself; the watchdog has no role in lock cleanup.

Use plane vs. management plane at runtime

The runtime makes the §1 use/management split concrete in a single dimension: does the process call core during its work? Use-plane processes do — Telegram gateway in every turn of its agent loop, the scheduler when it fires a registered entry, the external-agent endpoints when they handle a report or query. Their work generates signals because core drives the recorder. Management-plane processes do not — tool manager, soul manager, dream, watchdog, bootstrap, and inspection all operate directly on L2 libraries and adapters, never through core, and never produce signals as a side effect of their work. The same Typhoon binary serves both planes; which one a given process belongs to is determined by which subcommand was invoked.

One subtlety: when cron fires a management-plane subcommand (for example, the seeded typhoon dream entry), the cron-daemon side of the dispatch goes through core and a signal is recorded for the act of firing. The spawned dream subprocess then runs as a management-plane process and produces no further signals. So scheduled management activity is observable in the signal store as a single "fired" event, not as a stream of internal events.

Process supervision

Supervision in v0.1 is intentionally minimal. The watchdog does one thing — observe that the gateway and the scheduler are alive — and surfaces a failure when either is not. Failure surfacing is via stderr / syslog plus a non-zero exit on detected fault, so whichever process supervisor the host runs (systemd → journalctl, Docker → docker logs, runit, etc.) picks it up through its standard channel. The watchdog does not push alerts through the Telegram gateway: that would create a chicken-and-egg dependency where a dead gateway cannot be reported. If push-notification alerts become a v0.2 requirement, the watchdog gets its own outbound channel rather than borrowing the gateway's. Restart policy is the operator's responsibility — the watchdog reports, it does not act. Dream needs no supervision because it is not a daemon; cron decides when it runs and the file lock decides whether it can start. One-shot processes need no supervision because they are bounded and observable through their exit codes.

2.4 Subsystem view

The subsystem view groups the 15 modules from §2.2 into five coherent subsystems. The grouping is driven by cohesion of responsibility, not by layer: a subsystem may span L3 runtime modules and the L2 / L1 components they exclusively use. The five subsystems become the DLD's chapter partitioning — each one gets its own chapter, sized to be designed and reviewed independently. Cross-subsystem coupling is explicit and goes through the data-access subsystem in almost every case, which keeps the rest of the partitioning tractable.

S1 — Platform

The platform subsystem covers everything required to stand a Typhoon instance up, identify its users, and keep its daemons running. Bootstrap is one-shot — schema migrations, seed config, seed the operator as the first canonical user (with operator role), optional cloud-replica link, idempotent across re-runs. The platform also owns the operator-facing identity-binding management commands (list bindings, revoke a binding, mint an onboarding invite), which read and write through S5's Identity bindings library. The watchdog is continuous, observing the two long-running daemons (gateway and scheduler) and emitting failures through stderr / syslog. The DLD chapter for S1 covers the migration scheme (versioned, additive-only), the seed inventory (including the operator seed shape), the cloud-replica linking handshake, the binding management command grammar, the binding-verification mechanics (claim tokens, expiry, nonce), the watchdog's probe interval and failure-detection criteria, and the alerting-output format.

S2 — Channels & dispatch (use plane)

S2 is the entire use plane in one subsystem. It owns every path that brings external work into Typhoon (Telegram long-poll, external-agent one-shots, cron-fired entries), the core dispatcher that orchestrates that work, and the signal recorder that persists one signal per invocation. Two defining invariants are enforced here: "core is the only thing that drives the recorder," and "every invocation reaching core carries a resolved canonical_user_id." Gateways resolve identity via S5's Identity bindings before invoking core; the external-agent endpoints default to the operator. The DLD chapter for S2 covers the gateway protocol handling, the binding-resolution and onboarding flow for unbound peers, the agent loop's tool-dispatch sequence, the cron entry format and tick semantics, the external-agent endpoint contracts (input shape, exit codes, identity defaulting), the core's dispatch interface, and the recorder's write protocol (signal row tagged with canonical_user_id).

S3 — Self-growth

S3 is just the dream batch, but it is large and stands alone because of its scoring logic and its three-phase pipeline. Light, REM, and deep phases run sequentially; each emits structured outputs the next consumes. Dream reads from S5 (signal store, memory store, tool registry, existing proposals) and writes back through S5 (memory store, proposal queue) — it never calls into S2 or S4 directly. The file lock that makes dream a single-writer is part of this subsystem. The DLD chapter for S3 covers each phase's input/output contracts, the scoring formulas and weights, the LLM prompts for the deep phase, the soul-proposal generation rules, the stale-signal prune policy, and the cron-entry seeding.

S4 — Registry management

S4 owns the operator-driven advancement of proposals into the registry. Tool manager covers the unified typhoon tool namespace (proposal flow with embedded forge validators, plus full lifecycle operations on installed tools); every mutation gates on the operator role. Soul manager is the per-user flow for soul proposals — a user may approve their own, the operator may approve any — with atomic per-user config mutation on approval. Both managers query S5's Identity bindings for the role check, then compose against the proposal queue and the corresponding storage library (tool registry for one, config for the other), and both rely on the staging+rename atomicity pattern from §2.3 wherever a filesystem write is involved. The DLD chapter for S4 covers the proposal state-machine transitions, the validator implementations (sandbox tier check, path lint), the role-check policy at every subcommand, the operator-CLI command grammar and exit codes, the atomic-mutation transaction shapes, the lineage-and-history rules, the 3-strike rejection tracking (per pattern for cli proposals; per (canonical_user_id, config_key) for soul), and the promote-script (hand-written tool adoption) flow.

S5 — Data access

S5 is everything that knows about Typhoon's data model and everything that talks to external systems. The L2 libraries (tool registry, memory store, signal store, proposal queue, config, identity bindings) own data semantics — atomicity, retrieval bounds, success tagging, type validation, lineage rules, verified-binding semantics, the shared-vs-per-user data scoping, and role lookup. The L1 adapters (TursoDB / libSQL client with transaction primitives, filesystem ops, LLM clients for frontier and dream models, Telegram client, sandbox + path-lint helpers) own the protocols for talking to external systems. The L2 / L1 split inside this subsystem is the most important thing about it: domain semantics never leak into adapters; adapters never know about Typhoon-specific invariants. The DLD chapter for S5 covers the SQL schema (DDL — including the canonical-user table, binding table, and per-user vs system config split), each library's API surface (which APIs take canonical_user_id and which do not), the staging+rename pattern, transaction primitives, lock primitives, every adapter's external interface, and the swap-test criterion (replacing an adapter must not change anything above L1).

Subsystem dependencies

Almost every cross-subsystem dependency goes through S5. S2 and S4 produce mutations; S3 produces and consumes; S1 reads health. S5 is the integration point — that is the point of having a data-access subsystem.

Why this partitioning

Three properties make this set of subsystems suitable as DLD chapter boundaries.

  1. Each subsystem has a single headline responsibility that can be understood without referring to the others. Anyone reviewing the S3 chapter can do so without knowing the gateway protocol; anyone reviewing S2 can do so without knowing dream's scoring weights.
  2. Cross-subsystem coupling is bounded and explicit. S2 / S3 / S4 / S1 all depend on S5; S1 additionally observes S2's daemons; otherwise nothing crosses. This makes each chapter a small surface-area read.
  3. The runtime planes from §2.3 fall along these boundaries. S2 is the use plane in its entirety; S3, S4, and the operator-facing slice of S1 are management plane. The split this HLD treats as load-bearing remains visible at the DLD level rather than being smeared across chapters.

2.5 Project layout (workspace, crates, cross-cutting concerns)

The conceptual architecture from §2.1–§2.4 maps directly to a Rust workspace whose crate boundaries are the layer rule and the subsystem boundaries. Cargo's dependency check then enforces the rules at compile time — a forbidden upward call doesn't compile because the crate it would import is not in the consumer's Cargo.toml. The architecture stops being aspirational and becomes mechanical.

Five guarantees fall out of this:

  1. Layer rule = compile-time. Dispatch cannot reach an adapter directly because typhoon-adapters is not in typhoon-dispatch's Cargo.toml. Adding it would be a code-review-visible change to the manifest, not an accidental import.
  2. Subsystem boundary = compile-time. Registry-mgmt cannot accidentally call into dispatch for the same reason.
  3. No circular dependencies possible. Cargo refuses to compile cycles, so the layered DAG is mechanically guaranteed.
  4. Subsystem-scoped testing. cargo test -p typhoon-dream runs only S3 and its real deps. No false coupling to dispatch.
  5. DLD chapter ↔ crate is 1:1. Reviewers find code where the design said it would be.

Three rules govern when something becomes cross-cutting versus subsystem-local. (a) If two or more subsystem crates need it identically, it goes to typhoon-foundation. (b) If only one subsystem needs it, it stays inside that subsystem's crate. (c) If it touches an external system, it lives in typhoon-adapters regardless of how many subsystems consume it. The data-access layer (typhoon-data) is the only crate that bridges domain semantics with adapters; foundation never touches adapters.

Cross-cutting that this layout deliberately rejects

A few things commonly handled cross-cuttingly are not, on purpose. Dependency-injection containers — Rust's type system and explicit constructors are sufficient at this scale; a DI framework would obscure dependencies the layer rule wants visible. Aspect-oriented logging via macros that wrap every function — too magical; explicit tracing::instrument at module entry points stays readable. A monolithic shared or common crate that everyone imports — exactly the anti-pattern the layered split is meant to prevent; foundation stays narrow on purpose.

3. State diagrams

Three state machines survive in v0.1 and are made fully explicit here: the tool proposal status, the tool lifecycle, and the soul proposal status. Each diagram is the authoritative answer for "is this transition legal?" — implementations must reject any transition not drawn. There are no skill state machines; that vocabulary was deliberately removed (see PLAN §7 invariant 5).

Notation: rectangular states are stored values; the trailing [*] is process termination, not a value. Transition labels name the actor and the action. Guard conditions are written in square brackets. Side effects required to be in the same atomic transaction as the transition are written below the label in italics.

3.1 Tool proposal status (cli_proposals.status)

The tool proposal moves through a small linear state space. Approval emits a registry mutation that is atomic with the status flip (the staging+rename pattern from §2.3 spans this transaction). Rejection is the only branch that can recur; after the third rejection on the same pattern, dream is forbidden from drafting another proposal that originates from the same signal cluster.

stateDiagram-v2 [*] --> awaiting_forge : dream creates proposal awaiting_forge --> awaiting_user : tool propose submit [validators pass] awaiting_forge --> awaiting_forge : tool propose submit [path lint fails] awaiting_user --> approved : tool propose approve awaiting_user --> rejected : tool propose reject approved --> [*] rejected --> [*] : rejection_count < 3 rejected --> [*] : rejection_count = 3 (pattern blacklisted) note right of awaiting_user On submit: sandbox tier-claim check and path-lint run; tier silently downgraded if claim fails. end note note right of approved Atomic transaction: registry insert + binary install (stage then rename) + seed memory + status flip. end note note left of rejected Blacklist is enforced as a dream-side guard (count check before drafting), not as a state. end note

Invariants enforced by this machine:

  • The set {awaiting_forge, awaiting_user, approved, rejected} is total — no NULL, no other values. SQL CHECK is defense-in-depth.
  • The transition awaiting_user → approved is wrapped in BEGIN IMMEDIATE … COMMIT together with the registry insert, the seed-memory write, and (for replacements) the predecessor archival.
  • The 3-strike rule is enforced by checking SUM(rejection_count) for the originating pattern before dream creates a new proposal — not by a state on this machine. The blacklist is a guard on dream, not a status here.
  • A path-lint failure on submit does not advance state; the proposal stays awaiting_forge and the operator submits again with corrected source.
  • A sandbox tier-claim failure on submit advances state and silently downgrades the claimed tier — the operator reviews the source under stricter intensity.

3.2 Tool lifecycle

Once a proposal is approved, the tool itself enters a lifecycle covering installation, deactivation, replacement, deprecation, and removal. Every transition that touches a binary on disk uses the staging+rename pattern; every transition that affects more than one row is a single transaction.

stateDiagram-v2 [*] --> active : tool propose approve active --> disabled : tool disable disabled --> active : tool enable active --> superseded : tool propose approve [replacement] active --> active : tool rollback active --> deleted : tool delete disabled --> deleted : tool delete superseded --> deleted : tool delete deleted --> [*] : tool purge note right of active Steady state. Use and success counters increment here. Dream may propose replacement (rising error rate) or deprecation (prolonged disuse). tool rollback is a self-loop: active version swaps with a predecessor from .history/. end note note right of superseded Atomic swap on approve: predecessor moves to .history/, new binary installed via stage-then-rename, lineage and seed memory updated. end note note left of deleted Row retained for lineage and tool history; binary archived to .history/. purge is the irreversible terminal step that removes the row and clears .history/. end note note right of [*] Staging artifacts in ~/.typhoon/bin/.staging/ may persist if a process crashes between COMMIT and rename. The tool sync janitor sweeps them out of band; they are invisible to this state machine. end note

Invariants enforced by this machine:

  • Every transition that involves a filesystem operation is part of the same transaction as the registry mutation, via the staging+rename atomicity pattern from §2.3.
  • superseded is reachable only via approval of a proposal carrying replaces:<name>; same pattern, no auto-supersede.
  • tool rollback stays in active by definition — it swaps which version is the active binary, the state of the tool entity does not change.
  • deleted retains the registry row and the archived binary; purge is the irreversible terminal transition.
  • tool promote (operator adopts a hand-written script with no origin proposal) enters this machine at active directly, bypassing §3.1; it is recorded as approved_by='user' with no proposal pointer.

3.3 Soul proposal status (soul_proposals.status)

Soul proposals govern per-user config-row mutations originating from dream. The shape of the machine is the same as §3.1's tail: a single approval branch, a recurring rejection branch with the same 3-strike rule (here keyed per (canonical_user_id, config_key) — the rule is per user, so one user rejecting a soul change three times does not blacklist that key for other users). There is no forge step because the artifact is a config value, not a binary; therefore no awaiting_forge.

stateDiagram-v2 [*] --> pending : dream creates proposal pending --> approved : soul approve pending --> rejected : soul reject approved --> [*] rejected --> [*] : rejection_count < 3 rejected --> [*] : rejection_count = 3 (key blacklisted) note right of approved Atomic transaction: config row update + status flip in one step. end note note left of rejected Blacklist is per (canonical_user_id, config_key) and enforced as a dream-side guard, not a state. end note

Invariants enforced by this machine:

  • The set {pending, approved, rejected} is total — no NULL, no other values.
  • The transition pending → approved is wrapped in BEGIN IMMEDIATE … COMMIT together with the config table update — the value cannot change without the proposal closing in the same step, and the proposal cannot close without the value changing.
  • The 3-strike rule is enforced by checking SUM(rejection_count) for the same config_key before dream drafts a new soul proposal. As in §3.1, the blacklist is a guard on dream, not a state here.
  • There is no awaiting_forge equivalent: soul proposals carry the proposed value directly. Approval is one human gate, not two.

Cross-machine guarantees

Three guarantees span all three machines and belong here rather than in any one:

  1. Atomicity boundaries are state transitions. Every multi-step state change in §3.1 / §3.2 / §3.3 is one SQL transaction. Crashes between transitions leave the system in a defined state; crashes during a transition roll back to the prior state.
  2. 3-strike rules are dream-side guards, not states. The proposal machines have rejected as a normal terminal state; the "blacklist" is a count-based guard inside dream, checked before drafting. This keeps the state machines small and lets the operator reset a blacklist by clearing rejection counts in config if needed.
  3. No state machine carries an "in flight" or "processing" intermediate state. Transitions are instantaneous from the database's point of view (one transaction). If a process holding a transient lock dies, the lock is released and the source state stands.
