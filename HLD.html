1. System overview

Typhoon is a single Rust binary that observes an operator's work with an LLM agent, mines that work for repeated patterns, and routes high-value patterns through a forge loop that produces reusable command-line tools. Dream writes proposal briefs; the forge hardens each brief into a requirement and implementation; the operator ratifies. The runtime never synthesizes source itself.

The single-binary form is deliberate, but it does not mean a single process. One compiled artifact gives one install path, one upgrade story, one audit surface, and no client/server version skew — interactive commands, daemon entry points, scheduled dream runs, and proposal ratification all dispatch through the same compiled artifact. The subcommands share too much code (dream phases, signal capture, path lint, proposal flow) for splitting into multiple binaries to pay for itself in a single-user tool. This follows the git / docker / cargo model: one binary, many subcommands, daemon and one-shot modes side by side.

At runtime, multiple processes of that binary coexist, divided into two planes by whether they go through core.

The use plane covers every process whose work counts as Typhoon being used. Three sources feed into core. The channel gateway (typhoon gateway) is one long-running daemon that hosts channel-facing workflows for configured channels and forwards normalized requests to core. v0.1 configures a Telegram adapter; future channels (Slack, Discord, web hooks) add channel-adapter implementations and config entries, not one daemon per channel. The external-agent channel needs no gateway: each agent action is a one-shot subcommand invocation (typhoon signal record to report a tool call and outcome, typhoon memory query to pull retrieval context) that enters core directly through Typhoon's normal CLI dispatch. The scheduler (typhoon cron) is a general scheduler daemon. For use-plane targets, it fires entries through core — each entry is a cron expression plus a target, where the target is a user-facing typhoon subcommand or any tool on PATH. Core, invoked from any of these three sources, dispatches the actual work — calling the frontier LLM for channel-driven turns, spawning a tool subprocess, reading the memory store — and tells the signal recorder to persist a signal row for each invocation. The recorder is the only thing that touches the signal store. This is why scheduled use triggers flow through core: a scheduled deploy-status run is recorded exactly the way a channel-driven one would be, and dream can therefore learn from scheduled use.

The management plane bypasses core and operates directly on state. The unified tool manager (typhoon tool, covering both proposal-flow ingestion and lifecycle operations on the tool registry) is here. Dream (typhoon dream, a standalone file-locked batch job that drains signals into proposals and memories) is here. Health is daemon-scoped: only long-running gateway and cron processes need liveness state and optional heartbeats. One-shot CLI commands do not have health; they succeed or fail through their exit codes. Smaller operator one-shots (typhoon config, typhoon sql, typhoon dream stats, typhoon health) exist as CLI utilities for direct state inspection but are not architectural components in their own right. None of these generate signals — they are meta-operations on the registry, the config, the signal store itself. The scheduler dispatches differently depending on target. Use-plane targets (a tool on PATH, a user-facing typhoon subcommand) go through core and are recorded as signals exactly like channel-driven invocations — that is the whole point of routing scheduled use through core. Management-plane targets (the seeded typhoon dream entry, a periodic typhoon tool sync for janitor sweeps) are spawned by the scheduler directly without going through core, and produce no signals; their own operational tables (dream_runs, etc.) are how scheduled management activity is observed.

All processes share state through TursoDB transactions and filesystem locks regardless of plane. Process roles and concurrency rules are detailed in §2.3.

Typhoon's own command surface is agent-first. Every subcommand accepts and emits JSON, exits with structured codes, and is safe to compose into pipelines. An interactive operator and an external agent CLI invoke the same commands through the same paths — there is no human-only or agent-only mode. This is the precondition that makes the external-agent channel work without a separate API surface — the agent's one-shot subcommand invocations are the same calls the operator could type by hand.

This document uses these terms consistently. Typhoon's CLI is the typhoon binary's own command surface. Tools are the forged and operator-promoted artifacts installed into ~/.typhoon/bin/; "tool" subsumes both dream-forged CLIs and hand-written scripts the operator has promoted — both live in the same registry under the same rules. The tool manager is the typhoon tool subcommand namespace that ingests proposal deliveries and operates on the installed-tool registry; proposal rows are not registry rows until approval. Typhoon core is the in-binary module that dispatches every "use" invocation. The signal recorder is a separate component that persists signals to the signal store and is driven exclusively by core. The channel gateway is one long-running workflow daemon that polls configured channels, resolves identity, and feeds normalized requests to core. A channel adapter is a switchable provider implementation under that gateway; the Telegram adapter is the v0.1 instance and speaks to the external Telegram service. The external-agent channel does not need a gateway because each agent action is a fresh one-shot typhoon subcommand invocation that enters core directly. Forge is the operator-controlled implementation workflow that turns a dream proposal brief into hardened requirements, source, tests, and a correctness argument; in v0.1 it is manual, and in a later stage it may be assisted by a forge CLI.

Identity terms. A canonical user is the stable per-person identity inside Typhoon, referenced by canonical_user_id. Channels are external; canonical users are internal; a binding is a verified mapping from a channel-specific identity (e.g., a Telegram peer + bot account) to a canonical user. There are two roles: the operator ratifies proposals, manages tools, and runs administrative commands; a user sends signals and consumes memory and tools. v0.1 has exactly one operator — the deploying party, seeded at init — and zero or more additional users who join via channel binding.

Three architectural principles run through everything below. First, tools are the artifact — the tool registry (TursoDB rows + binaries in ~/.typhoon/bin/, reconciled by checksum and filesystem locks) is the system's central artifact store, and every other piece of the lifecycle exists to grow it safely. The registry is fronted by a shared access module — a library, not a process — so both core (for runtime tool lookup during dispatch) and the tool manager CLI (for operator-driven CRUD) read and write through one place; schema knowledge and registry invariants live in this module rather than being re-implemented per consumer. Second, core dispatches; the recorder records; only core can drive the recorder — every process that uses Typhoon (the channel gateway, the external-agent endpoints, scheduled use triggers) routes through core, and only core tells the signal recorder to write. The recorder is a separate component, the sole writer to the signal store; gateways and the scheduler adapt or trigger but never invoke the recorder directly. Anything that bypasses core produces no signal. This split — use plane through core, management plane around it — is the structuring rule for the runtime and shows up directly in §2.

Third, tools are shared; per-user data is isolated. A single tool registry serves every canonical user — when one user's signals motivate a tool, every user benefits once the operator approves. Signals, memory, and soul (per-user config) are scoped per canonical_user_id; cross-user reads are a privacy bug. Storage shape is one shared store per concept (one signals table, one memories table, one config table), with rows tagged by canonical_user_id; per-user isolation is enforced by state APIs that take the user ID and filter every read and write — no per-user queues or partitioned tables. Dream is the deliberate exception: its signal scan reads across all users so cross-user pattern overlap can become evidence for a shared tool. The operator role is the gate on tool ratification and registry mutations; user-role processes contribute signals and consume memory and tools, nothing more. This per-user / shared split lives in S5's data-access libraries (§2.4) and is checked at gateway-resolution and management-command entry; the rest of the architecture is identity-agnostic.

The tool manager is firmly on the management plane: it operates directly on proposals and the registry and never goes through core, because managing tools is meta, not use. Dream is also management plane — it is a standalone batch process that reads accumulated signals and writes proposals and memories. The two are complements: core feeds the signal table by recording use; dream drains it into the proposal queue; the tool manager ratifies approved results into the registry.

Typhoon's lifecycle has four stages. Stage 1 runs once for an instance. Stages 2 through 4 repeat: online interaction records signals, dream organizes those signals into proposal briefs, and forge plus ratification turns selected briefs into installed CLIs or rejected proposals. After ratification, the system returns to online interaction.

Stage 1 — Cold start

The operator initialises a Typhoon instance against an operator-provided TursoDB cloud database with typhoon init --url URL --token TOK. Schema migrations and seed run together and are idempotent: re-running the bootstrap on an already-initialised database is a no-op. The seed creates the operator as the first canonical user and marks them with the operator role; this is the only canonical user that exists at end of bootstrap. After bootstrap the runtime is dormant — no daemon runs, no signals exist. One Typhoon runtime instance owns one TursoDB database that serves multiple users (the operator plus any users who later join via channel binding). The same cloud database may be reached from other machines for operator review or forge workflows, but running a second runtime writer instance against that database is not supported.

Stage 2 — Online interaction

The operator opens a session through one of two channels. The Telegram channel goes through the channel gateway daemon (with the Telegram adapter); the external-agent channel uses no gateway at all and enters core directly via one-shot CLI invocations. Both paths feed into the same core, so core sees a uniform interface regardless of source.

Telegram channel (typhoon gateway with the Telegram adapter enabled). The channel gateway is one daemon that hosts the agent loop for configured channels. For Telegram, the gateway polls through the Telegram adapter; the adapter speaks Telegram protocol and returns provider-neutral channel updates. On every incoming message, the gateway resolves the Telegram peer (channel + bot account + peer ID) to a canonical user via the binding store. A bound peer flows straight into the loop. An unbound peer goes through onboarding — the binding flow's exact shape is a DLD detail; the HLD only requires that a message is never accepted from an unverified peer. Once the canonical user is known, every step routes through core: the gateway asks core for that user's relevant memories, sends the prompt plus available tools to the frontier LLM, hands any tool calls the LLM emits to core for execution (shell, filesystem, tools on PATH), feeds the results back to the LLM, loops until the LLM stops, and posts the final reply back through the Telegram adapter. As core dispatches each invocation, it tells the signal recorder to write a signal row tagged with that user's canonical_user_id. In this path Typhoon is the agent runtime; core is the dispatcher under it.

External-agent channel (Claude Code, Cursor, Codex, …). The operator runs the agent on their machine; the agent hosts its own LLM client, its own tool dispatcher, and its own conversation loop. Typhoon never sees the frontier LLM. The agent uses Typhoon by invoking subcommands directly — typhoon memory query for retrieval context, typhoon signal record to report a tool call and outcome. In v0.1 only the operator uses this channel, so each invocation defaults to the operator's canonical_user_id with no explicit identity argument; when a second user wants this channel later, an explicit --user flag will be added. Each invocation is a one-shot process that enters core through Typhoon's normal CLI dispatch; core then drives the recorder (or reads memory) just as it would for the Telegram-channel path. A signal row is a signal row regardless of which source delivered the call. In this path the agent runtime lives outside Typhoon, but channel data still flows into core and through the recorder into the signal store.

This pattern generalizes — adding a future channel (Slack, Discord, web hooks) means writing a new gateway process that drives core through the same interface. Core does not change, the signal model does not change, and the management plane is unaffected.

After a turn closes (in either channel), a success flag is stamped onto the signal chain by a simple post-hoc rule: clean exit plus no corrective next turn. Signals accumulate; nothing else happens online.

Stage 3 — Dream

The dream cycle is triggered by the scheduler spawning the seeded management-plane entry or by the operator invoking it directly. Once dream starts, it runs on the management plane: it operates directly on state without going back through core, since reading signals and writing proposals is meta and would loop if it generated more signals. A file lock guarantees a single dream runs at a time. Three phases run sequentially against the accumulated signals: a light phase deduplicates and sorts; a REM phase clusters successful signal chains, detects repeated workflows, and checks each candidate against the existing tool registry to flag collisions and replacement candidates; a deep phase, using a cheap batch LLM, emits three kinds of output — memories that land directly in the memory store, CLI proposal briefs that enter the proposal queue with status awaiting_forge, and soul proposals that enter their own queue. Stale signals that never graduated are pruned. Dream is the canonical scheduled job in v0.1, but it shares the scheduler mechanism with any other entry the operator adds.

Stage 4 — Forge and ratify (out-of-band first, automated later)

The operator reads awaiting_forge proposal briefs via the tool manager (typhoon tool propose list / show) and drives a forge workflow to produce a CLI delivery. In v0.1, forge is manual: the operator chooses the coding agent or environment (Codex, Claude Code CLI, Cursor, or similar), sharpens the brief into a high-quality requirement, asks the forge to implement the source, iterates on tests and examples, and prepares a correctness argument. Typhoon does not invoke the forge, does not constrain the language, and does not judge the source's correctness. A later stage may add a forge CLI that automates part of this handoff, but that CLI is support tooling around the same contract rather than a new source of truth.

The operator submits the delivery back through the tool manager with typhoon tool propose submit <id> --requirements <file> --source <file> [--tests <file>]. On submission, Typhoon attaches the hardened requirement, source, optional tests, declared dependencies, and forge correctness argument to the existing cli_proposals row, then runs the hardcoded-path lint. A failed path lint is fatal and bounces the proposal. Tier is not mechanically verified in v0.1; Dream's rough tier and the forge's revised tier are reviewed by the operator from the brief, requirement, source, dependencies, and correctness argument. On pass the proposal advances to awaiting_user. No tool-registry row is created at submit time; unfinished or unapproved forge work is proposal data, not an installed CLI.

The operator reviews the proposal brief, hardened requirement, delivery, tests, and forge correctness argument together via the tool manager (typhoon tool propose approve | reject). Approval is the first point where a CLI enters the tool registry: the registry row is inserted, the tool binary is installed, a seed memory is written so the agent can discover the new tool, and — for a replacement — the predecessor is moved into .history/. The DB part is protected by a TursoDB transaction; the filesystem part is protected by the registry mutation protocol in §2.3. Soul proposals follow the same pattern with one substitution: the artifact is a config value rather than a tool binary, and the atomic write updates the row instead of the filesystem. A pattern rejected three times stops dream from re-proposing it.

After approval, the next lifecycle step is simply more online interaction. The installed CLI is shared through the registry, surfaced by memory retrieval, invoked as an ordinary subprocess on PATH, and measured by use/success counters. Those counters become future signals for Dream: a rising error rate can lead to a replacement proposal, and prolonged disuse can lead to a deprecation proposal. Operator-driven tool management is not a separate lifecycle stage; it is the management surface available throughout the loop for proposal review, installed-tool inspection, disable/enable, rollback, delete, purge, promote, dependency checks, and sync.

The tool manager (typhoon tool) is the operator's direct, agent-independent control surface for proposal ingestion and the installed-tool registry, covering the entire lifecycle through one unified namespace. Every mutating typhoon tool subcommand requires the operator role; user-role processes can use installed tools but cannot mutate proposals or the registry. The ingestion half (tool propose list / show / submit / approve / reject) is what stage 4 invokes. list, show, and submit operate on cli_proposals; only approve creates or updates a registry row. The operations half acts on already-installed tools: read-only listing, inspection, and lineage (tool list with filters by tier or staleness, tool show for source/stats/origin, tool diff and tool history for version comparison) and state-changing commands (tool disable takes a tool off PATH while keeping the registry row; tool enable reverses it; tool rollback restores the previous version from .history/; tool delete marks the registry row deleted (retained for lineage and tool history) and archives the binary; tool purge hard-deletes the registry row and clears .history/; tool promote adopts a hand-written script the operator authored outside the dream loop and adds it as approved_by='user' with no origin proposal; tool check-deps scans declared external dependencies across the registry; tool sync reconciles installed binaries from approved registry rows on the current machine). Every installed-tool state change uses the registry mutation protocol so the DB row, checksum, and on-disk executable can be reconciled. Broader inspection paths are read-only: typhoon sql for SELECT queries, typhoon dream stats for the proposal funnel, and typhoon health for gateway/cron liveness. typhoon sql accepts SELECT only; writes (INSERT / UPDATE / DELETE / DDL) are hard-rejected so the inspection path can never accidentally mutate state.

2. Architecture

Purpose: describe Typhoon's structure from three complementary angles, then group the pieces into subsystems that partition the DLD.

2.1 Layer view

The layer view uses the common architecture vocabulary a reader would expect from a layered application. It is a stack view: each row is a floor, and a floor may depend only on the row directly below it. It is not a request-flow diagram, a crate dependency map, or a module catalog. It deliberately avoids concrete technology, concrete commands, and concrete subsystems. The integration layer is split into two blocks: service adapters for non-storage external services, and data access for Typhoon state. External systems are shown as a side column and attach only through the service-adapter block; they are not the data-store layer. Foundation is shown as a vertical side column because it supplies narrow primitives to the stack, but it is not itself a request path or a business floor. The static dependency view is intentionally not specified here; it belongs in the DLD, where exact crate edges and manifest rules can be reviewed against implementation.

Presentation / interaction layer. Converts the outside world's events into internal requests and renders responses back out. It validates that a request is well-formed and extracts identity claims. It does not own use-case decisions or durable work.

Application / core layer. Owns Typhoon's use cases and orchestration. It coordinates the use plane and management plane, sequences work, validates workflow preconditions, and invokes the integration layer. It does not write stores directly and does not speak external service protocols directly.

Integration layer. Contains two blocks with different responsibilities. The service-adapter block reaches non-storage external services. The data-access block reaches Typhoon's data stores. Keeping both blocks in one layer says they are both boundary code below application, while keeping them separate says service calls and state persistence are not the same concern.

Service adapter block. Owns request/response mapping, retries, and protocol-specific error translation for non-storage services, covering both outbound calls and inbound long-poll / webhook fetches. It is the only block that talks to the external-systems side column; it does not write the data store.

Data access block. Exposes typed state APIs to the floors above. It owns schema shape, query boundaries, transaction composition, coordination primitives, and store-specific error translation. It does not decide user intent or own use-case orchestration; it persists and retrieves facts according to decisions made above it. It is the only block that talks to the data store layer.

Data store layer. The deepest layer contains the database and artifact storage that hold Typhoon state. It is reached only through the data-access layer. This is the normal database/storage floor of the architecture, not an "external systems" bucket.

External systems side column. Contains non-storage systems Typhoon may call through service adapters. The current examples are channel APIs and LLM APIs. External systems do not attach to presentation, application, or data store directly in this layer view.

Foundation sidecar. Provides narrow primitives used everywhere, such as identity values, time, errors, and observability. It is a side concern rather than a business floor and must stay small; it should never become a general-purpose "shared" bucket.

Five boundaries are load-bearing. First, interaction stays thin — translate and render, do not decide. Second, application owns workflow decisions — use-case decisions live with the use cases that apply them, not at the edge. Third, data access is the only path to stores — upper layers never write the database or artifact storage directly. Fourth, service adapters are the only path to external services — upper layers never speak channel or LLM wire protocols directly. Fifth, data stores hold facts; data access owns how facts are read and written. Foundation's rule is separate: stay narrow. The concrete static dependency view that enforces these rules in code is a DLD responsibility, not an HLD diagram.

2.2 Logical view (modules)

The logical view enumerates the modules — function groups — that make up Typhoon. Each module has one job, named clearly, with explicit consumers and producers. The catalog maps modules to their primary layer from §2.1, but the layer labels are descriptive, not crate names. Data stores are not listed as modules because they are storage infrastructure; data-access modules are the Typhoon-side typed APIs that read and write them. Service adapters are also shown here when a module needs an outside capability such as a channel API or LLM API. Modules marked Interaction + application currently straddle two floors because their boundary surface and use-case logic are described together at HLD level; the DLD decides whether those concerns split into separate implementation units. 2.3 Process view

The process view is about runtime topology: what OS processes exist, which run continuously, which come and go, what they share, and how concurrency is bounded. Coordination — transactions and the dream lock

Three mechanisms hold the runtime together: SQL transactions for database state changes, per-tool filesystem locks for installed-tool mutations, and the dream lock for the one batch job that must not run twice.

Transactions. Every state change that affects more than one database row is wrapped in BEGIN IMMEDIATE … COMMIT with rollback on any database failure. Operations that also touch the filesystem use the registry mutation protocol below so the DB row and executable can be reconciled deterministically. The atomic DB operations:

OperationWhat is committed atomically
Tool approval (new)registry insert + seed memory write + proposal status → approved; filesystem install follows the registry mutation protocol
Tool approval (replacement)registry lineage update + seed memory update + proposal status → approved; predecessor archival and new binary install follow the registry mutation protocol
Tool rollbackregistry active-version update; binary swap follows the registry mutation protocol
Tool disable / enable / delete / purgeregistry status mutation; binary move/restore/delete follows the registry mutation protocol
Soul approvalconfig row update + proposal status → approved
Forge submissionrequirements + source + optional tests attach to proposal + path lint + proposal status → awaiting_user

SQL transactions serialize database state, but they do not make filesystem writes transactional. If two processes write the same path without coordination, the filesystem gives last-writer-wins behavior; it will not know which write matches the committed registry row. Typhoon therefore treats installed-tool mutation as a registry protocol, not a raw file copy. The normal mutation path is:

  1. Start a TursoDB write transaction with BEGIN IMMEDIATE.
  2. Acquire the per-tool filesystem lock.
  3. Write the executable to a staging path outside the active PATH.
  4. Verify permissions and checksum.
  5. Write the registry, seed-memory, and proposal-status rows inside the open DB transaction.
  6. Perform an atomic rename(2) into ~/.typhoon/bin/<name>.
  7. Commit the DB transaction, then release the filesystem lock.

If a normal pre-commit step or commit fails while the process is alive, the DB transaction rolls back and Typhoon removes the staging/final file before releasing the lock. A crash can still leave filesystem leftovers because there is no real distributed transaction between TursoDB and the host filesystem. v0.1 chooses the DB as source of truth: no CLI is considered installed unless an approved registry row exists and its checksum matches the file on disk. tool sync removes unreferenced staging/final files and surfaces any active-row/checksum mismatch for operator repair. If a future version requires a hard PATH-level guarantee that no orphan executable can ever appear, the install layout should move to stable shims or a content-addressed artifact store instead of direct final-path writes.

Dream lock. The dream batch acquires ~/.typhoon/dream.lock via filesystem locking before doing any work. A second invocation while the first holds the lock fails fast with a lock-error exit code. This guards both operator-invoked dream and cron-fired dream — the lock does not care who tried to start the second instance. The lock is process-scoped (released on process exit, even on crash), so a stale lock from a dead PID resolves itself; daemon health has no role in lock cleanup.

Use plane vs. management plane at runtime

The runtime makes the §1 use/management split concrete in a single dimension: does the process call core during its work? Use-plane processes do — channel gateway in every turn of its agent loop, the scheduler when it fires a use-plane entry, the external-agent endpoints when they handle a report or query. Their work generates signals because core drives the recorder. Management-plane processes do not — tool manager, soul manager, dream, bootstrap, health, and inspection all operate directly on application/data-access modules, never through core, and never produce signals as a side effect of their work. The same Typhoon binary serves both planes; which one a given process belongs to is determined by which subcommand was invoked.

One subtlety: when cron fires a management-plane subcommand (for example, the seeded typhoon dream entry), it spawns that subcommand directly rather than routing through core. No signal is recorded for the act of firing. Scheduled management activity is observed through its own operational tables such as dream_runs and through daemon health, not through the signal store.

Process supervision

Supervision in v0.1 is intentionally minimal. Only the gateway and scheduler are daemons, so only they carry daemon health state and optional heartbeat timestamps. typhoon health is a read-only one-shot that reports whether those daemon states are fresh, along with DB latency and recent recorder/dream status. It does not restart processes and it does not create health records for ordinary CLI invocations. Restart policy is the operator's responsibility through the host supervisor (systemd, Docker, runit, etc.). Dream needs no daemon supervision because it is transient; cron decides when it runs and the file lock decides whether it can start. One-shot processes need no supervision because they are bounded and observable through their exit codes.

2.4 Subsystem view

The subsystem view groups the modules from §2.2 into five coherent subsystems. The grouping is driven by cohesion of responsibility, not by physical crate layers: a subsystem may include interaction, application, data-access, storage-adapter, and service-adapter responsibilities that belong together. The five subsystems become the DLD's chapter partitioning — each one gets its own chapter, sized to be designed and reviewed independently. Cross-subsystem coupling is explicit and goes through the data-access and adapters subsystem in almost every case, which keeps the rest of the partitioning tractable.

S1 — Platform

The platform subsystem covers everything required to stand a Typhoon instance up, identify its users, and keep daemons observable. Bootstrap is one-shot — connect to the operator-provided TursoDB URL/token, run schema migrations, seed config, and seed the operator as the first canonical user with operator role; re-runs are idempotent. The platform also owns the operator-facing identity-binding management commands (list bindings, revoke a binding, mint an onboarding invite), which read and write through S5's Identity bindings library. typhoon health is a read-only one-shot that reports gateway and scheduler liveness/heartbeat state, DB latency, and recent recorder/dream status. The DLD chapter for S1 covers the migration scheme (versioned, additive-only), the seed inventory (including the operator seed shape), the binding management command grammar, the binding-verification mechanics (claim tokens, expiry, nonce), the daemon heartbeat rows, and the health output format.

S2 — Channels & dispatch (use plane)

S2 is the entire use plane in one subsystem. It owns every path that brings external work into Typhoon (channel gateway turns, external-agent one-shots, cron-fired use entries), the core dispatcher that orchestrates that work, and the signal recorder that persists one signal per invocation. Two defining invariants are enforced here: "core is the only thing that drives the recorder," and "every invocation reaching core carries a resolved canonical_user_id." The channel gateway resolves identity via S5's Identity bindings before invoking core; the external-agent endpoints default to the operator. The DLD chapter for S2 covers the gateway workflow, the binding-resolution and onboarding flow for unbound peers, the agent loop's tool-dispatch sequence, the cron entry format and tick semantics, the external-agent endpoint contracts (input shape, exit codes, identity defaulting), the core's dispatch interface, and the recorder's write protocol (signal row tagged with canonical_user_id).

S3 — Self-growth

S3 is just the dream batch, but it is large and stands alone because of its scoring logic and its three-phase pipeline. Light, REM, and deep phases run sequentially; each emits structured outputs the next consumes. Dream reads from S5 (signal store, memory store, tool registry, existing proposals) and writes back through S5 (memory store, proposal queue) — it never calls into S2 or S4 directly. The single-writer policy belongs to S3; S5 provides the filesystem lock primitive. The DLD chapter for S3 covers each phase's input/output contracts, the scoring formulas and weights, the LLM prompts for the deep phase, the soul-proposal generation rules, the stale-signal prune policy, and the cron-entry seeding.

S4 — Registry management

S4 owns the operator-driven advancement of proposals into the registry. Tool manager covers the unified typhoon tool namespace (proposal flow with required forge delivery fields and path lint, plus full lifecycle operations on installed tools); every mutation gates on the operator role. Proposal submit attaches forge output to cli_proposals; approval is the first operation that mutates the installed-tool registry. Soul manager is the per-user flow for soul proposals — a user may approve their own, the operator may approve any — with atomic per-user config mutation on approval. Both managers query S5's Identity bindings for the role check, then compose against the proposal queue and the corresponding storage library (tool registry for one, config for the other), and both rely on the registry mutation protocol from §2.3 wherever a filesystem write is involved. The DLD chapter for S4 covers the proposal state-machine transitions, path-lint implementation, the role-check policy at every subcommand, the operator-CLI command grammar and exit codes, the atomic-mutation transaction shapes, the lineage-and-history rules, the 3-strike rejection tracking (per pattern for cli proposals; per (canonical_user_id, config_key) for soul), and the promote-script (hand-written tool adoption) flow.

S5 — Data access & adapters

S5 is the lower implementation subsystem that contains Typhoon's data-access libraries and typed adapters. Data-access libraries (tool registry, memory store, signal store, proposal queue, config, identity bindings) own persistence mechanics and typed state APIs — atomicity, retrieval bounds, success fields, type validation, lineage persistence, verified-binding rows, the shared-vs-per-user data scoping, and role lookup. Storage adapters own driver mechanics for the data-store layer. Service adapters own the protocols for non-storage services such as channel APIs and LLM APIs. The important boundary is that application code uses typed APIs and adapters; it does not embed SQL, filesystem driver details, channel wire formats, or LLM provider request formats. The DLD chapter for S5 covers the SQL schema (DDL — including the canonical-user table, binding table, and per-user vs system config split), each library's API surface (which APIs take canonical_user_id and which do not), the registry mutation protocol, transaction primitives, lock primitives, every adapter's external interface, and the exact static dependency view that enforces the boundary in code.

Because S5 is intentionally broad, the DLD may split its chapter by component group: data-access APIs, storage adapters, service adapters, and transaction/lock primitives.

Subsystem dependencies

Almost every cross-subsystem dependency goes through S5. S2 and S4 produce mutations; S3 produces and consumes; S1 reads health. S5 is the lower access point — that is the point of having a data-access and adapters subsystem.

Why this partitioning

Four properties make this set of subsystems suitable as DLD chapter boundaries.

  1. Each subsystem has a single headline responsibility that can be understood without referring to the others. Anyone reviewing the S3 chapter can do so without knowing the gateway protocol; anyone reviewing S2 can do so without knowing dream's scoring weights.
  2. Cross-subsystem coupling is bounded and explicit. S2 / S3 / S4 / S1 all depend on S5; S1 additionally observes S2's daemons; otherwise nothing crosses. This makes each chapter a small surface-area read.
  3. The runtime planes from §2.3 fall along these boundaries. S2 is the use plane in its entirety; S3, S4, and the operator-facing slice of S1 are management plane. The split this HLD treats as load-bearing remains visible at the DLD level rather than being smeared across chapters.
  4. Process-view participants have clear owners without becoming new subsystems. Sequence participants from §2.3 map either to a top-level subsystem-owned component group, to S5's access/adapters boundary, or to an external actor/service. This keeps the subsystem count manageable while still giving DLD authors a concrete hierarchy.

The partition above is a candidate shape, not a frozen dependency contract. The DLD must either ratify it or replace it with a better physical design while preserving the HLD boundaries in §2.1–§2.4.

Static dependency view — DLD handoff

The static dependency view is deliberately deferred to detailed design. The DLD should define the exact crate dependency DAG and prove these guarantees mechanically:

  1. Responsibility boundary = compile-time where practical. Application crates should not bypass data-access APIs to write stores directly, and should not embed service wire-format details unless the DLD explicitly accepts and documents an exception.
  2. Subsystem boundary = compile-time where practical. Registry management should not accidentally call into dispatch, and dream should not depend on registry-management workflow code.
  3. No circular dependencies possible. Cargo refuses to compile cycles, so the dependency DAG is mechanically guaranteed.
  4. Subsystem-scoped testing. cargo test -p typhoon-dream runs only S3 and its real deps. No false coupling to dispatch.
  5. DLD chapter ↔ implementation owner is clear. Reviewers find code where the design said it would be, even if a chapter maps to more than one crate.

Cross-cutting concerns

Cross-cutting concerns are everything every crate may need but nothing should reimplement. They have one of three homes; each home has a strict rule.

ConcernHomeRule
Error types (one root error enum per subsystem; conversions to/from)typhoon-foundation defines the trait + common variants; subsystem crates define their own thin enum that From-convertsNo anyhow in library crates; only the binary may use anyhow. Subsystem errors carry context, not just text.
Clock / time (now, sleep, deadline)typhoon-foundation::clock — trait + production impl + test impl in typhoon-testkitNo direct calls to std::time::SystemTime::now or tokio::time::sleep outside foundation. Everything injectable.
ID generation (proposal IDs, run IDs, signal IDs, idempotency keys, canonical_user_id)typhoon-foundation::idsOne generator type per ID kind (newtype). No raw Uuid at API surfaces. canonical_user_id is a newtype to prevent string mix-ups across module boundaries.
Identity types (canonical_user_id, Role enum, binding tuple)typhoon-foundation defines the types; typhoon-data defines the binding library; typhoon-dispatch resolves bindings at gateway / endpoint entry; typhoon-registry-mgmt checks role at every mutationAuthorization is checked at boundaries (gateway, endpoint, management subcommand), not deep inside libraries; libraries trust the canonical_user_id they receive.
Logging (structured, level-controlled)typhoon-foundation re-exports a logging facade (e.g., tracing); only the binary configures the subscriberLibrary crates emit only; never configure. Log fields use a small fixed vocabulary (subsystem, run_id, signal_id, …).
Tracing / spans (request IDs, dispatch chains)Same — tracing spans, started at interface entry and at core dispatchEvery signal carries the originating span ID for post-hoc correlation.
Telemetry / metrics (counters, gauges, histograms)typhoon-foundation::metrics — façade onlyLibrary crates increment counters via the façade; the binary chooses an exporter (or none, in v0.1).
Feature flags / config knobsAlready in S5 Config (typhoon-data::config); foundation defines no flagsAll knobs live in the config table. No compile-time cargo features for behaviour, only for optional adapter integrations.
Panic hook + crash reportingtyphoon-foundation::panic_hookInstalled once by the binary at startup. Logs the panic with the run / span ID before propagating.
Async runtimeChosen and started by the binary; passed via handles or context where neededLibrary crates do not own runtime startup. Tokio is the likely default for daemons, but the runtime choice remains binary-level.
Test fixtures, fakes, builderstyphoon-testkit — fake clock, in-memory adapter fakes, proposal builders, signal-chain buildersProduction crates never depend on testkit; testkit is [dev-dependencies] everywhere it's used.
Code style, lints, formattingWorkspace-level rustfmt.toml, clippy.toml, deny-by-default lint set in Cargo.tomlSame rules apply to every crate; no per-crate exceptions without a review-visible #[allow].

Three rules govern when something becomes cross-cutting versus subsystem-local. (a) If two or more subsystem crates need it identically, it goes to typhoon-foundation. (b) If only one subsystem needs it, it stays inside that subsystem's crate. (c) If it is a storage driver or service client, it lives in typhoon-adapters regardless of how many subsystems consume it. The S5 data-access and adapters subsystem bridges Typhoon semantics with stores and service clients; foundation never touches adapters.

Cross-cutting that this layout deliberately rejects

A few things commonly handled cross-cuttingly are not, on purpose. Dependency-injection containers — Rust's type system and explicit constructors are sufficient at this scale; a DI framework would obscure dependencies the DLD should make visible. Aspect-oriented logging via macros that wrap every function — too magical; explicit tracing::instrument at module entry points stays readable. A monolithic shared or common crate that everyone imports — exactly the anti-pattern the foundation sidecar is meant to prevent; foundation stays narrow on purpose.

3. State diagrams

Three state machines survive in v0.1 and are made fully explicit here: the tool proposal status, the tool lifecycle, and the soul proposal status. Each diagram is the authoritative answer for "is this transition legal?" — implementations must reject any transition not drawn. There are no skill state machines; that vocabulary was deliberately removed (see PLAN §7 invariant 5).

Notation: rectangular states are stored values; the trailing [*] is process termination, not a value. Transition labels name the actor and the action. Guard conditions are written in square brackets. Side effects that share the same database transition, or the same registry mutation protocol, are written below the label in italics.

3.1 Tool proposal status (cli_proposals.status)

The tool proposal moves through a small linear state space. Approval emits a registry mutation and closes the proposal in the same database transaction; any binary install or archival follows the registry mutation protocol from §2.3. Rejection is the only branch that can recur; after the third rejection on the same pattern, dream is forbidden from drafting another proposal that originates from the same signal cluster.

stateDiagram-v2 [*] --> awaiting_forge : dream creates proposal brief awaiting_forge --> awaiting_user : tool propose submit [requirements + source + path lint pass] awaiting_forge --> awaiting_forge : tool propose submit [path lint fails] awaiting_user --> approved : tool propose approve awaiting_user --> rejected : tool propose reject approved --> [*] rejected --> [*] : rejection_count < 3 rejected --> [*] : rejection_count = 3 (pattern blacklisted) note right of awaiting_user On submit: attach hardened requirement, source, optional tests, dependencies, and correctness argument. end note note right of approved DB transaction: registry insert + seed memory + proposal status flip. Binary install follows registry mutation protocol. end note note left of rejected Blacklist is enforced as a dream-side guard (count check before drafting), not as a state. end note

Invariants enforced by this machine:

  • The set {awaiting_forge, awaiting_user, approved, rejected} is total — no NULL, no other values. SQL CHECK is defense-in-depth.
  • The transition awaiting_user → approved is wrapped in BEGIN IMMEDIATE … COMMIT together with the registry insert/update, seed-memory write, and proposal status flip. Filesystem install or archival follows the registry mutation protocol from §2.3.
  • The 3-strike rule is enforced by checking SUM(rejection_count) for the originating pattern before dream creates a new proposal — not by a state on this machine. The blacklist is a guard on dream, not a status here.
  • A path-lint failure on submit does not advance state; the proposal stays awaiting_forge and the operator submits again with corrected source.
  • Proposal submit never creates or updates a tool-registry row; registry mutation starts only at approval.

3.2 Tool lifecycle

Once a proposal is approved, the tool itself enters a lifecycle covering installation, deactivation, replacement, deprecation, and removal. Every transition that touches a binary on disk follows the registry mutation protocol; every transition that affects more than one database row is a single SQL transaction.

stateDiagram-v2 [*] --> active : tool propose approve [*] --> active : tool promote active --> disabled : tool disable disabled --> active : tool enable active --> superseded : tool propose approve [replacement] active --> active : tool rollback active --> deleted : tool delete disabled --> deleted : tool delete superseded --> deleted : tool delete deleted --> [*] : tool purge note right of active Steady state. Use and success counters increment here. Dream may propose replacement (rising error rate) or deprecation (prolonged disuse). tool rollback is a self-loop: active version swaps with a predecessor from .history/. end note note right of superseded Atomic swap on approve: predecessor moves to .history/, new binary installed via registry mutation protocol; lineage and seed memory updated. end note note left of deleted Row retained for lineage and tool history; binary archived to .history/. purge is the irreversible terminal step that removes the row and clears .history/. end note note right of [*] Filesystem leftovers may persist if a process crashes during install. The DB registry row and checksum are source of truth; tool sync removes or surfaces mismatches out of band. end note

Invariants enforced by this machine:

  • Every transition that involves a filesystem operation follows the registry mutation protocol from §2.3: TursoDB transaction, per-tool filesystem lock, staging path, checksum, atomic rename, and sync-based repair of crash leftovers.
  • superseded is reachable only via approval of a proposal carrying replaces:<name>; same pattern, no auto-supersede.
  • tool rollback stays in active by definition — it swaps which version is the active binary, the state of the tool entity does not change.
  • deleted retains the registry row and the archived binary; purge is the irreversible terminal transition.
  • tool promote (operator adopts a hand-written script with no origin proposal) enters this machine at active directly, bypassing §3.1; it is recorded as approved_by='user' with no proposal pointer.

3.3 Soul proposal status (soul_proposals.status)

Soul proposals govern per-user config-row mutations originating from dream. The shape of the machine is the same as §3.1's tail: a single approval branch, a recurring rejection branch with the same 3-strike rule (here keyed per (canonical_user_id, config_key) — the rule is per user, so one user rejecting a soul change three times does not blacklist that key for other users). There is no forge step because the artifact is a config value, not a binary; therefore no awaiting_forge.

stateDiagram-v2 [*] --> pending : dream creates proposal pending --> approved : soul approve pending --> rejected : soul reject approved --> [*] rejected --> [*] : rejection_count < 3 rejected --> [*] : rejection_count = 3 (key blacklisted) note right of approved Atomic transaction: config row update + status flip in one step. end note note left of rejected Blacklist is per (canonical_user_id, config_key) and enforced as a dream-side guard, not a state. end note

Invariants enforced by this machine:

  • The set {pending, approved, rejected} is total — no NULL, no other values.
  • The transition pending → approved is wrapped in BEGIN IMMEDIATE … COMMIT together with the config table update — the value cannot change without the proposal closing in the same step, and the proposal cannot close without the value changing.
  • The 3-strike rule is enforced by checking SUM(rejection_count) for the same config_key before dream drafts a new soul proposal. As in §3.1, the blacklist is a guard on dream, not a state here.
  • There is no awaiting_forge equivalent: soul proposals carry the proposed value directly. Approval is one human gate, not two.

Cross-machine guarantees

Three guarantees span all three machines and belong here rather than in any one:

  1. Atomicity boundaries are state transitions. Every multi-row database state change in §3.1 / §3.2 / §3.3 is one SQL transaction. Filesystem side effects are reconciled against the DB source of truth by the registry mutation protocol. Crashes between transitions leave the database in a defined state; crash leftovers on disk are repaired or surfaced by tool sync.
  2. 3-strike rules are dream-side guards, not states. The proposal machines have rejected as a normal terminal state; the "blacklist" is a count-based guard inside dream, checked before drafting. This keeps the state machines small and lets the operator reset a blacklist by clearing rejection counts in config if needed.
  3. No state machine carries an "in flight" or "processing" intermediate state. Transitions are instantaneous from the database's point of view (one transaction). If a process holding a transient lock dies, the lock is released and the source state stands.
